Why mundlejs?

Lots of javascript loading up-front (concatenated) = long wait-time for initial page load.
Loading all javascript asynchronously, as needed= too many requests resulting in poor performance and poor user experience.
Manually optimizing concatenated scripts for deployment= a pain, and an insolvable probable for dynamic sites.

Symptom of all these problems: client-side code which is not truly modular and reusable, the way our server-side code is.

What does it look like?

  • For a step in the workflow where the user would expect a "loading" delay for the dependency: require('./someModule.js',myCallbackFunc);

  • For dependencies that you want to be available immediately: someModuleExports = require('./someModule.js');

Your code does what it says- no processing before deployment to make it mean something different, and no fake synchronicity that gets covered up by a build step. Synchronous require in your client-side code means that module will be available immediately, asynchronous means you can arrange to keep the client busy while waiting for the module to load.

How does it work?

The secret is that loading client-side modules is not a client only problem, a server-side component should exist and work hand-in-hand with the client. What the server-side does is parse synchronous dependencies (the second form in the "what?" section) and package them all together so they will be available immediately.

Benefits:

  • Client-side code can be as modular as you like, without affecting real-life performance.
  • Really fast- caching of bundles is included, so it's faster than express even at serving single files
  • Very intuitive- when your code means what it says, everyone benefits
  • Build coffeescript, jade, etc. for the client on-the-fly, with cacheing.
